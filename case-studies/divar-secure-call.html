Solving Harassment at Scale

How I Protected 1.2M Users on Iran's Largest Marketplace

The Context

What is Divar Classified Ads Platform?

Divar is Iran's largest and most popular online classified ads platform (launched in 2012). It serves as a go-to marketplace for everyday transactions, covering categories like real estate, vehicles, jobs, services, and personal items.

53M+
Weekly buyers & sellers
180M+

72B+
Annual Ad Views
The Issue

Customer care centres were flooded with harassment reports in the Clothing & Apparel category.

  • Sellers were deleting listings or churning entirely, fearing repeat incidents.
  • Even after victims deleted listings, calls continued because their numbers were already exposed.
  • Divar risked losing a category that anchored daily engagement.

"It was infuriating. I took down my Divar ad hoping to stop the calls, but they kept coming. Someone told me my number was in a Telegram group."

Layla, 32
Professional artist selling bespoke clothing
Success Metrics

Before designing anything, we defined what success looked like, and where we could not compromise.

North star: Reduce harassment-tagged calls from the call centre.

Non-negotiable guardrails
  • Must not reduce legitimate buyer-seller call volume
  • Must not harm seller trust or increase listing abandonment
  • Must work on older Android devices (47% of the user base at the time)
  • Must not increase app size significantly
  • Must not expose real phone numbers at any point in the flow
Constraints & Tradeoffs

Six constraints shaped every decision before a single wireframe was drawn.

1
Older device compatibility

VoIP was an obvious solution, but 47% of users were on older Android versions where VoIP quality was inconsistent and implementation would significantly increase app size. Rejected early.

2
Trust erosion risk from hiding numbers

Research showed that sellers who hid phone numbers received fewer responses from legitimate buyers. Any solution that made sellers feel less reachable would backfire.

3
The anonymity problem

Hiding numbers entirely attracted a new problem: scammers could operate without accountability. The solution needed to protect sellers without enabling bad actors.

4
No ground truth on how harassment was actually happening

The call centre knew that harassment was occurring but not how phone numbers were being obtained. My first instinct turned out to be only part of the story.

5
56% pilot failure rate

The first implementation failed hard. The constraints I learned from that failure shaped every subsequent iteration.

6
Timeline pressure

The category was losing sellers in real time. Speed mattered, but a bad fast solution was worse than a slower right one.

Digging to Find the Real Problem

I interviewed harassed sellers and analysed behavioural data to understand:

  1. How did this happen?
  2. How were users currently trying to solve or prevent it?

Investigation A: The Social Media Link
User Quote

"One caller even knew my eye colour. They clearly found my WhatsApp through the phone number on Divar."

Sara, 26
Fashion retail worker

Assumption:

Users are being harassed through their phone numbers being cross-referenced on social media platforms.

Validation:

I sampled 1,000 fashion sellers and compared WhatsApp profile characteristics between a random sample and known harassment victims. I classified them into three types:

A. Phone number with WhatsApp account + profile image
B. Phone number with WhatsApp account, without profile image
C. Phone number with no WhatsApp account

% of sellers by WhatsApp account type

Random Sample
Victims of Harassment
■ WA account + profile image ■ WA account, no image ■ No WA account
SQL
Show query
-- WhatsApp profile type: random sample vs harassment victims WITH seller_sample AS ( SELECT s.user_id, CASE WHEN w.has_account = TRUE AND w.has_profile_image = TRUE THEN 'WA + image' WHEN w.has_account = TRUE AND w.has_profile_image = FALSE THEN 'WA, no image' ELSE 'No WA account' END AS wa_type, CASE WHEN h.user_id IS NOT NULL THEN 'Victims of Harassment' ELSE 'Random Sample' END AS cohort FROM sellers s LEFT JOIN whatsapp_profiles w ON s.phone_number = w.phone_number LEFT JOIN harassment_reports h ON s.user_id = h.reported_user_id WHERE s.category_name = 'Fashion/Apparel' AND s.created_at >= CURRENT_DATE - INTERVAL '2 months' ) SELECT cohort, wa_type, COUNT(*) * 100.0 / SUM(COUNT(*)) OVER (PARTITION BY cohort) AS pct FROM seller_sample GROUP BY cohort, wa_type ORDER BY cohort, wa_type; -- Note: Table and column names are changed for company privacy
Key Findings
  • Harassment victims were significantly more likely to have a recognisable profile image on social media.

We could not recommend sellers hide profile pictures, since this reduces trust with legitimate customers.

Validation: Experience-Based Behaviour

I examined posting patterns over the past two months, comparing:

  • First-time sellers: % showing phone numbers in their initial ads
  • Experienced sellers: % showing phone numbers in subsequent ads (2nd, 3rd listings)

% ads with hidden phone number over listing sequence

SQL
Show query
-- Experience-based posting behaviour analysis WITH user_ads_ranked AS ( SELECT user_id, ad_id, category_name, phone_number_visible, created_at, ROW_NUMBER() OVER ( PARTITION BY user_id ORDER BY created_at ) AS ad_sequence_number FROM advertisements WHERE created_at >= CURRENT_DATE - INTERVAL '2 months' ) SELECT CASE WHEN ad_sequence_number = 1 THEN '1st Ad' WHEN ad_sequence_number = 2 THEN '2nd Ad' WHEN ad_sequence_number = 3 THEN '3rd Ad' ELSE '4th+ Ad' END AS ad_number, COUNT(CASE WHEN category_name = 'Fashion/Apparel' AND phone_number_visible = FALSE THEN 1 END) * 100.0 / NULLIF(COUNT(CASE WHEN category_name = 'Fashion/Apparel' THEN 1 END), 0) AS fashion_pct_hidden, COUNT(CASE WHEN category_name != 'Fashion/Apparel' AND phone_number_visible = FALSE THEN 1 END) * 100.0 / NULLIF(COUNT(CASE WHEN category_name != 'Fashion/Apparel' THEN 1 END), 0) AS other_pct_hidden FROM user_ads_ranked GROUP BY CASE WHEN ad_sequence_number = 1 THEN '1st Ad' WHEN ad_sequence_number = 2 THEN '2nd Ad' WHEN ad_sequence_number = 3 THEN '3rd Ad' ELSE '4th+ Ad' END ORDER BY MIN(ad_sequence_number); -- Note: Table and column names are changed for company privacy
Key Findings
  • Most experienced fashion sellers hide their phone numbers, at a higher rate than in any other category.
  • Sellers learn defensive behaviour immediately, with a sharp decrease between 1st and 2nd ad in the clothing category.
  • Data shows fashion sellers have been forced to adopt systematic self-protection strategies.

And we know what people do to prevent it: They use in-app messaging instead of sharing their numbers.

Pick the Right Solution

We evaluated three potential approaches to solve the privacy issue.

Option 1
✕ Rejected
User Education on Harasser Tactics

Informing users about how harassers typically identify them.

  • Could damage Divar's reputation
  • Might discourage users from posting
  • Sophisticated harassers would adapt
Option 2
✕ Rejected
In-App VoIP Calls

Voice over Internet Protocol calls within the app, making user phone numbers invisible.

  • Expensive to scale
  • Inconsistent internet quality across Iran
  • 47% of users on older Android couldn't use it
  • Increased app size significantly
Option 3
✓ Chosen
Intermediary Numbers for Secure Calls

When a buyer clicks "view contact number," display a Divar intermediary number instead of the seller's real number. Calls are forwarded behind the scenes.

The Implementation Journey

The First Attempt

We ran a 3-hour pilot in the clothing category. Nothing visible changed: instead of showing the post publisher's number, we showed a Divar proxy number and diverted calls from the ad viewer to the post publisher.

The results were… problematic.

56%
Calls Failed to Connect
Initial 3-hour pilot result

Root Cause Analysis

I developed assumptions for the 56% failure rate:

User Side Issues
  • Sellers couldn't see buyer numbers for follow-up contact
  • Users blocked the intermediary number thinking it was spam
  • All callers showed the same number, causing massive confusion
Technical Side Issues
  • Call volume peaks were much higher than expected
  • System bottlenecks during high traffic
Business Side Issues
  • Calls lasted longer than predicted, inflating costs
  • The anonymity attracted potential scammers

Refined Solutions

🔄
Bidirectional Mapping

Sellers could now call buyers back, creating two-way communication.

⏱
Extended Mapping Time

Increased from 20 seconds to 15 minutes for flexibility in communication.

🔊
Voice Message Introduction

Added explanation at call start: "This is a secure call from Divar…", which guides users and deters scammers.

🏗
Number Pool Solution

Created 100 intermediary numbers. Each buyer-seller pair got a unique number. Maintained mapping for repeat contacts. Prevented permanent blocking issues.

The Impact

After rollout, harassment reports dropped dramatically and sellers felt safe enough to keep listings live.

Quantitative Results

  • ~60% reduction in harassment reports within 8 weeks of launch.
  • ~1.2 million distinct users per week continuously protected by the secure call system.
  • Significant increase in secure call adoption in the clothing category.

Qualitative Results

  • Users felt safer posting clothing ads, and sellers who previously churned returned.
  • Increased trust in platform protection among the community.
  • Maintained communication efficiency without compromising privacy.
What I'd Do Differently
Run the pilot longer before drawing conclusions

The 3-hour pilot gave directional signal but limited confidence. A longer pilot with a holdout group would have provided cleaner data on which failure modes were systemic versus time-of-day artefacts.

Instrument earlier

I had to build the data queries reactively, after the pilot failed. Designing the measurement framework before launch would have let me catch failure modes faster in subsequent iterations.

Key Learnings
1
The obvious problem isn't always the real problem

Initial instinct was to hide numbers. The real issue was the connection between phone numbers and social media profiles.

2
Users are already solving problems, so listen to them

The solution came from observing how experienced users adapted: avoiding phone numbers, relying on in-app chat.

3
Failure is a feature, not a bug

The 56% failure rate in our pilot gave us exactly the insights we needed to build a robust solution.

4
Iterate ruthlessly

From immediate fixes (voice messages) to structural improvements (number pools), each iteration solved specific user pain points.

Reflection

This project reinforced my belief that solving hard problems starts with understanding behaviour. By blending research, data, and rapid iteration, we built a safety layer that still protects millions of people each week. The harassment problem may never disappear entirely, but Divar is significantly safer for the users who need it most.

Special thanks to the product manager who kept this project grounded, and to the entire dev team.

Omid Morousi
